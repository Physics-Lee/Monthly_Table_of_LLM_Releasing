# LLM 芯片层 + 云计算层 月表预研

## 核心问题

如果把芯片层和云计算层也纳入现有月表体系，会是什么样的？

---

## 一、数据结构对比

### 现有模型层（已落地）

| 维度 | 模型层 | 说明 |
|---|---|---|
| **频率** | 高 | 每月 5-15 个模型发布 |
| **粒度** | 细 | 具体到模型名称和版本 |
| **时效性** | 即时 | 发布当月记录 |
| **厂商数** | 32 家 | 每家一列 |

### 芯片层（预研）

| 维度 | 芯片层 | 说明 |
|---|---|---|
| **频率** | 低 | 每年 3-8 个重大发布 |
| **粒度** | 粗 | 架构代际、产品系列 |
| **时效性** | 滞后 | 发布 → 量产 → 部署，跨度数月 |
| **厂商数** | 10-15 家 | NVIDIA、AMD、Google、Intel、云厂商自研、 startups |

**关键差异**：
- 芯片不是"每月都有新鲜事"
- 同一芯片的发布/量产/部署是三个不同时间点
- 制程节点（3nm/4nm）和架构（Hopper/Blackwell）是跨年度大事件

### 云计算层（预研）

| 维度 | 云计算层 | 说明 |
|---|---|---|
| **频率** | 中 | 每月 2-5 个重大发布 |
| **粒度** | 混合 | 服务发布、定价变更、区域扩展 |
| **时效性** | 即时 | 但持续性更新多 |
| **厂商数** | 8-10 家 | 三大云 + 二线云 + AI 云初创 |

**关键差异**：
- 云服务有"发布"和"GA（通用可用）"两个时间点
- 定价变更也是重要事件
- 区域扩展（如 AWS 新 Region）影响可用性

## 二、芯片层月表设计方案

### 方案 A：独立表格（推荐）

**文件**：`ai_chip_timeline_2020-2026.csv`

**列设计**：
```
Year-Month, NVIDIA, AMD, Google-TPU, Intel, Amazon-Trainium, Microsoft-Maia, Meta-MTIA, Huawei-Ascend, Qualcomm, Startup-Others
```

**内容示例**：

| Year-Month | NVIDIA | AMD | Google-TPU | ... |
|---|---|---|---|---|
| 2022-03 | H100 发布 | | | |
| 2022-09 | H100 量产 | | | |
| 2023-03 | | | TPU v4 大规模部署 | |
| 2023-11 | H200 宣布 | | | |
| 2023-12 | | MI300X 发布 | TPU v5p 发布 | |
| 2024-03 | Blackwell B200 发布 | | | |
| 2024-08 | H200 上市 | | | |

| 2025-06 | | MI350X 发布 | | |
| 2026-01 | | | | Maia 200 发布 |

**优点**：
- 不干扰现有模型表
- 芯片事件稀疏，独立表格避免大量空列
- 可以追溯 2020 年（H100 之前）

**缺点**：
- 与模型表分开，关联性弱
- 需要单独维护

### 方案 B：合并到现有表格（不推荐）

在现有 34 列后追加芯片列：
```
... , NVIDIA-Chip, AMD-Chip, Google-Chip, Other-Chips
```

**问题**：
- 现有表格 42 行 × 34 列已经很宽
- 芯片事件密度低，95% 单元格为空
- 芯片和模型在同一行但无直接关联

## 四、三层关联设计

### 理想状态：三层互相引用

```
模型层（Model）→ 训练所用芯片（Chip）→ 部署所用云（Cloud）

示例：
GPT-4 (2023-03)
  └─ 训练芯片：H100 (2022-09 量产)
  └─ 部署云：Azure OpenAI Service (2022-11 宣布)

Claude 3 (2024-03)
  └─ 训练芯片：H100 + TPU v5p
  └─ 部署云：AWS Bedrock + Google Cloud
```

**实现方式**：
- 在模型表中添加 `Training-Chip` 和 `Deploy-Cloud` 两列
- 用芯片表和云表的外键引用
- 但信息往往不透明（训练细节保密）

## 五、实施建议

### Phase 1：独立表格（立即可做）

1. **芯片表** `ai_chip_timeline.csv`
   - 15 家厂商 × 80 个月（2020-2026）
   - 预计填充率：15-20%（稀疏但关键）

2. **云表** `ai_cloud_timeline.csv`
   - 10 家厂商 × 80 个月
   - 预计填充率：25-30%

### Phase 2：关联标注（远期）

在模型表中添加可选列：
```
... , Training-Chip, Deploy-Cloud, Notes
```
用于标注已知信息（如公开披露的训练基础设施）

### Phase 3：可视化层（最远）

- 三层叠加时间线
- 芯片发布 → 模型训练 → 云服务上线 的因果关系链
- 例如：H100 量产 (2022-09) → GPT-4 训练 → GPT-4 发布 (2023-03)

## 六、数据质量挑战

### 芯片层
1. **保密性**：训练所用芯片往往不公开
2. **时间模糊**："发布"≠"量产"≠"部署"
3. **版本复杂**：H100 SXM5 vs PCIe，不同 SKU

### 云层层
1. **持续性更新**：服务是渐进式改进，无明确"发布日"
2. **区域差异**：同一服务在不同 Region 上线时间不同
3. **定价复杂**：按需/预留/Spot，不同实例类型

## 八、结论与建议

### 是否加入月表？

**建议：加入，但用独立表格**

理由：
1. 芯片和云是 LLM 生态的底层支撑，有记录价值
2. 与模型层数据结构差异大，不宜硬塞进同一 CSV
3. 独立表格可以各自扩展（芯片可追溯到 2020，云可记录定价）

### 优先级

| 优先级 | 内容 | 原因 |
|---|---|---|
| P0 | 芯片表（ai_chip_timeline） | 事件少但关键，容易维护 |
| P1 | 云表（ai_cloud_timeline） | 事件多，但价值高 |
| P2 | 三层关联 | 需要大量调研，远期目标 |
