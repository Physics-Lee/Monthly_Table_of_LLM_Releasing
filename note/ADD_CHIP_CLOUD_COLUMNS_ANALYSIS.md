# 在现有 CSV/MD 中直接添加"芯片"和"云计算"两列的可行性分析

## 现有表格状况

- **42 行**（2022-11 至 2026-04）
- **34 列**（Month + 32 厂商 + LLM-Applications）
- **填充率**：模型层约 60-70%，LLM-Applications 约 25%

## 直接添加两列的问题

### 1. 密度严重不匹配 ❌

| 列 | 月均事件数 | 填充率估算 |
|---|---|---|
| 模型厂商（32列） | 5-15 个/月 | 60-70% |
| LLM-Applications | 2-4 个/月 | 25% |
| **芯片（新增）** | **0.5-1 个/月** | **~10%** |
| **云计算（新增）** | **1-2 个/月** | **~15%** |

**后果**：
- 36 列的表格，底部两列 85-90% 为空
- 视觉上像"两张瘸腿"挂在表格末端
- 打印/PDF 导出时浪费大量空白

### 2. 语义不一致 ❌

现有表格的列语义：
```
每列 = 一个厂商，每格 = 该厂商当月发布的模型
```

如果加"芯片"和"云计算"列：
```
芯片列 = 当月所有芯片厂商的事件集合（NVIDIA + AMD + Google + ...）
云计算列 = 当月所有云厂商的事件集合（Azure + AWS + GCP + ...）
```

**问题**：
- 其他列是"单厂商"，这两列是"多厂商汇总"
- 其他列格式是"模型名"，这两列格式是"厂商 产品名"（如"NVIDIA H200"）
- 链接生成逻辑完全不同：模型列用 `inferURL(modelName)`，芯片/云列需要 `inferURL(vendor + modelName)`

### 3. 超宽表格 ❌

- 36 列在桌面端还能看，但已接近极限
- 手机端：现有 34 列已经需要横向滚动，再加两列体验更差
- Markdown 表格在 GitHub 上渲染时，36 列可能导致换行混乱

### 4. 维护复杂度 ❌

**现有工作流**：
```
每月更新：查各厂商模型发布 → 填入对应列 → 自动 build
```

**新增后工作流**：
```
每月更新：
  - 查各厂商模型发布 → 填入对应列
  - 查所有芯片厂商事件 → 汇总填入"芯片"列
  - 查所有云厂商事件 → 汇总填入"云计算"列
  - 处理混合格式的链接映射
  - build（需要修改 inferURL 支持厂商前缀）
```

**问题**：
- 芯片事件季度性出现，容易遗漏
- 云事件类型多样（服务发布/降价/Region扩展/融资），标准不统一
- 同一单元格多个事件时，分隔符和链接处理复杂

### 5. 构建脚本改造 ❌

现有 `build-json.js` 假设每列 = 一个厂商：
```javascript
const vendors = headers.slice(1); // 去掉 Month
vendors.forEach(vendor => { ... });
```

如果加"芯片"和"云计算"：
```javascript
// 需要特殊处理这两列
if (vendor === '芯片') {
  // 解析单元格内容，拆分多个事件
  // 为每个事件找链接（需要厂商前缀）
} else if (vendor === '云计算') {
  // 同上
} else {
  // 现有逻辑
}
```

脚本复杂度显著增加。

## 如果一定要做，实现方案

### CSV 格式

```csv
Month,OpenAI,...,LLM-Applications,芯片,云计算
2022-11,,,,,
2022-12,,,Perplexity,,
2023-03,ChatGPT-4,Cursor,AMD MI300X | Google TPU v5p,Azure OpenAI GA
2024-03,Claude 3,,NVIDIA Blackwell B200,AWS Trainium2 GA
```

**注意**：
- 芯片列用 `|` 分隔多个事件
- 事件格式：`厂商 产品名`（如"NVIDIA H200"）
- 云计算列同理

### MD 表格效果

```markdown
| Month | ... | LLM-Applications | 芯片 | 云计算 |
|---|---|---|---|---|
| 2022-11 | ... | | | |
| 2022-12 | ... | Perplexity | | |
| 2023-03 | ... | Cursor | [AMD MI300X](url) + [Google TPU v5p](url) | [Azure OpenAI GA](url) |
```

### build-json.js 修改点

1. **inferURL 增强**：支持 `厂商 产品名` 格式匹配
2. **链接映射**：为芯片/云事件添加链接规则
3. **单元格解析**：支持 `+` 或 `|` 分隔多事件

## 我的建议：不要做 ❌

### 推荐替代方案

#### 方案 A：独立子表格（最佳）

在同一网页中，模型表下方增加两个可折叠区域：

```
[📊 LLM 模型发布时间表] ← 现有表格
[📁 芯片/半导体大事记]  ← 可展开
[☁️ 云计算/基础设施大事记] ← 可展开
```

**优点**：
- 保持模型表整洁
- 芯片/云用更适合的格式（事件列表而非月表）
- 可独立维护

#### 方案 B：每月备注栏（折中）

在现有表格中加一列"月度备注"，偶尔标注重大芯片/云事件：

```csv
Month,...,备注
2023-03,...,TPU v5p 发布；Azure OpenAI GA
```

**优点**：不破坏表格结构
**缺点**：信息不完整，只是点缀

#### 方案 C：独立页面（长远）

三个独立页面/标签页：
- `index.html` → 模型表
- `chips.html` → 芯片时间线
- `cloud.html` → 云时间线

**优点**：每层都有合适的展示形式
**缺点**：工作量最大

---

## 结论

| 方案 | 可行性 | 维护难度 | 用户体验 | 推荐度 |
|---|---|---|---|---|
| 直接加两列 | 技术上可行 | 高 | 差（太宽、太空） | ⭐ |
| 独立子表格 | 容易实现 | 低 | 好 | ⭐⭐⭐⭐⭐ |
| 月度备注栏 | 最容易 | 最低 | 一般 | ⭐⭐⭐ |
| 独立页面 | 中等 | 中 | 最好 | ⭐⭐⭐⭐ |

**我的立场**：不建议直接加列。芯片/云的数据特征与模型表差异太大，硬塞会破坏现有表格的可用性。

**如果你坚持要做**：我可以做，但需要：
1. 接受 36 列超宽表格
2. 接受 85%+ 的空单元格
3. 修改 build 脚本支持混合格式
4. 重新定义单元格内容格式（厂商前缀）

你决定。
